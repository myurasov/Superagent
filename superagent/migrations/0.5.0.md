## Summary

Marker migration for the `0.5.0` release that lands **first-class Claude
Code support** alongside Cursor. The release adds: a `CLAUDE.md` shim
that `@`-imports `AGENTS.md`; `.claude/settings.json` wiring the
`UserPromptSubmit` hook to `tools/log_user_query.py --source claude-code`;
`.claudeignore` to keep Claude Code from dual-loading `.cursor/`;
per-IDE MCP starting-point templates `.mcp.json.claude` and
`.cursor/mcp.json.cursor` (content-identical — Superagent has no
per-IDE OAuth `client_id` constraint, unlike NVIDIA-internal frameworks);
the `superagent/tools/ide.py` env-driven detection helper; init.md
steps 4½ + 4¾ for IDE-shim verification and MCP-template bootstrap;
and rewritten AGENTS.md "IDE setup" + "Prompt-cache discipline"
sections that name Cursor and Claude Code symmetrically.

None of the workspace data layout, memory file schemas, or required
config keys change. Existing workspaces keep working without
modification. Users who run only Cursor or only Claude Code are
unaffected; users who run both (the common case for this maintainer)
get matching tooling support.

   `tools/version.py.set_workspace_version(workspace, "0.5.0")`. The
   `migrate` skill does this automatically after this migration's
   `Validate` step succeeds.

## Validate

- [ ] `workspace/.version` reads `0.5.0`.
- [ ] `workspace/_memory/config.yaml` is still readable as YAML.
- [ ] No other files under `workspace/` were modified by this

## Revert

1. Write `workspace/.version` back to `0.4.0`. Implementation:
   `tools/version.py.set_workspace_version(workspace, "0.4.0")`.

After revert, no other workspace state needs unwinding — the release
adds framework files (under `superagent/` and at the repo root) and
updates docs; nothing under `workspace/` is touched.

## Notes

- The legacy `.mcp.json.example` template is renamed to `.mcp.json.claude`
  in this release. If a user's workspace shell history references
  `.mcp.json.example`, it will resolve to a missing file going
  forward — direct the user at `.mcp.json.claude` (Claude Code
  starting point) or `.cursor/mcp.json.cursor` (Cursor starting
  point) depending on which IDE they use.
- The two MCP runtime files (`.mcp.json` and `.cursor/mcp.json`) are
  intentionally NOT touched by this migration. The `init` skill's
  step 4¾ handles the template-to-runtime copy on next run; if a
  user previously kept `.cursor/mcp.json` as a symlink to
  `.mcp.json` (pre-`0.5.0` workaround), the next `init` run detects
  the legacy state and offers to replace it with a regular-file
  copy. The migration itself does not touch the repo-root `.cursor/`
  tree.
- IDE detection runs from environment variables on every call (no
  sticky `preferences.ide` config). Users switching between Cursor
  and Claude Code interchangeably get the right answer on every
  turn without touching `_memory/config.yaml`.
