## Summary

Introduces the `_memory/email/` per-message archive backing
`contracts/email-capture.md`. Every email the agent reads via
`mcp_user-gmail_read_email` or sends via `mcp_user-gmail_send_email`
will mirror through `superagent/tools/email/archive.py` into this tree.

This migration only creates the empty scaffold (`_index.yaml` with the
initial counter shape and an empty `_messages.jsonl`); the per-message
JSONs and the lazy `attachments/` subdirectory are produced on first
real capture. Independent of the pre-existing dormant Gmail ingestor at
`_memory/_gmail/`.

## Pre-flight checks

- [ ] `workspace/_memory/` exists.
- [ ] `workspace/.version` reads `0.3.0`.
- [ ] `workspace/_memory/email/_index.yaml` does NOT already exist (the
      migration would otherwise be a no-op; the skill should have
      skipped this step).

## Migrate

1. Create the directory: `mkdir -p workspace/_memory/email`.

2. Write `workspace/_memory/email/_index.yaml` containing exactly:

   ```yaml
   schema_version: 1
   attachments: metadata
   counts:
     total: 0
     full: 0
     stubs: 0
     inbound: 0
     outbound: 0
     attachments_saved: 0
   first_capture: null
   last_capture: null
   ```

3. Create the empty sidecar: `touch workspace/_memory/email/_messages.jsonl`.

## Validate

- [ ] `workspace/_memory/email/` is a directory.
- [ ] `workspace/_memory/email/_index.yaml` parses as YAML and contains
      `schema_version: 1`, `attachments: metadata`, and a `counts`
      mapping with the six expected keys.
- [ ] `workspace/_memory/email/_messages.jsonl` exists and is empty.
- [ ] `workspace/_memory/email/attachments/` does NOT exist yet (the
      attachments folder is lazy — created on first `save_attachment`
      call by `superagent.tools.email.archive`).
- [ ] `uv run python -m superagent.tools.email.archive --workspace
      <path> stats` runs cleanly and prints `total: 0`.

## Revert

1. Remove the empty sidecar: `rm workspace/_memory/email/_messages.jsonl`
   (use `rm`, not `rm -rf` — if the file is non-empty, the migration is
   NOT safe to revert and the user must resolve manually).
2. Remove the index: `rm workspace/_memory/email/_index.yaml`.
3. Remove the directory: `rmdir workspace/_memory/email` (use `rmdir`,
   not `rm -rf` — if the directory has captured messages, the migration
   is NOT safe to revert and the user must resolve manually).

After revert, no email archive scaffold exists; the helpers in
`superagent/tools/email/archive.py` re-create it lazily on first use,
so reverting only matters when the user wants to genuinely uninstall
the feature.
