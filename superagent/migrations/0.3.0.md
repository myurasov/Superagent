## Summary

Introduces the Gmail ingestor (`superagent/tools/ingest/gmail.py`) and
seeds its row in `workspace/_memory/data-sources.yaml`. The row lands
with `enabled: false` so nothing runs without the user's say-so. Also
creates the empty output directory `workspace/_memory/_gmail/` where the
ingestor will write monthly JSONL shards on first run.

The ingestor is **read-only at the skill layer** (only calls
`messages.list` + `messages.get`), even though OAuth scopes granted by
the gongrzhe Gmail MCP are wider (`gmail.modify` +
`gmail.settings.basic`). The framework's "no remote write" rule
(`AGENTS.md` § "Privacy and data location") enforces this.

After migrating, the user enables the row and triggers the first ingest
manually:

```bash
# Inspect the row that the migration just added.
grep -A 12 '^  - id: "gmail"$' workspace/_memory/data-sources.yaml

# When ready to enable + pull:
#   1. Edit data-sources.yaml: set gmail.enabled to true.
#   2. Run a dry-run probe + ingest:
uv run python -m superagent.tools.ingest.gmail --probe
uv run python -m superagent.tools.ingest.gmail --dry-run
uv run python -m superagent.tools.ingest.gmail
```

## Pre-flight checks

- [ ] `workspace/_memory/data-sources.yaml` exists and parses as YAML.
- [ ] No row with `id: "gmail"` already exists in
      `data-sources.yaml.sources` (would indicate a partial prior
      migration; abort and resolve manually).
- [ ] `workspace/.version` reads `0.2.0` (the from-version this
      migration expects).

## Migrate

1. Read `workspace/_memory/data-sources.yaml`. Append the following row
   to the `sources:` list (PRESERVE all comments + the placeholder
   row above; this is an append, not a rewrite):

   ```yaml
     - id: "gmail"
       kind: "api"
       enabled: false
       capture_mode: "manual"
       schedule: "manual"
       recency_window_days: 30
       backfill_window_days: 365
       max_items_per_run: 200
       writes_upstream: false
       labels: ["INBOX", "SENT"]
       auth:
         kind: "oauth"
         ref: "~/.gmail-mcp/credentials.json"
         scope:
           - "https://www.googleapis.com/auth/gmail.modify"
           - "https://www.googleapis.com/auth/gmail.settings.basic"
         server_id: "gmail"
       last_ingest: null
       last_run: null
       failure_streak: 0
       notes: |
         Headless ingest uses the Gmail API directly with OAuth tokens
         stored at ~/.gmail-mcp/credentials.json by the gongrzhe Gmail
         MCP. Chat-time ad-hoc Gmail tools are served by the same MCP
         (configured separately in ~/.cursor/mcp.json). See
         superagent/docs/data-sources.md § gmail for the setup writeup.
   ```

2. Create the empty output directory: `mkdir -p workspace/_memory/_gmail`.

## Validate

- [ ] `data-sources.yaml.sources` contains exactly one row with
      `id == "gmail"`.
- [ ] That row has `enabled: false`, `kind: "api"`,
      `writes_upstream: false`.
- [ ] `workspace/_memory/_gmail/` is a directory (and is empty for a
      fresh migration).
- [ ] `uv run python -m superagent.tools.ingest.gmail --probe` returns
      either `available` (auth set up; user is ready to enable) or
      `needs_setup` (no creds yet; user follows the docs to set them
      up). Both are acceptable post-migration outcomes; the migration
      itself does not require working Gmail auth.

## Revert

1. Read `workspace/_memory/data-sources.yaml`. Remove the row whose
   `id == "gmail"`. Preserve every other row + every comment.
2. Remove the empty output directory: `rmdir workspace/_memory/_gmail`
   (use `rmdir`, not `rm -rf` -- if the directory has user data in it,
   the migration is NOT safe to revert and the user must resolve
   manually).

After revert, the `gmail` row is gone and any local cache of pulled
messages is preserved on the filesystem only if the directory was
non-empty (in which case the rmdir step fails safe and the user
chooses what to do with the data).
