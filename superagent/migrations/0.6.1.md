## Summary

Adds the `archive_hook.py` bridge that wires the three Gmail MCP tool
calls (`send_email`, `read_email`, `search_emails`) to the existing
`tools/email/archive.py` capture helpers via the IDE `PostToolUse` hook
surface. Updates `.claude/settings.json` and `.cursor/hooks.json` with
three matchers each so capture-on-touch now happens automatically under
both Claude Code and Cursor, without depending on per-call agent
discipline. `contracts/email-capture.md` documents the new wiring.

This migration touches workspace data **only by bumping `.version`**.
The archive layout under `workspace/_memory/email/` is unchanged from
0.6.0; new entries land there organically as the user reads or sends
Gmail under the new hook wiring.

## Revert

1. Write `workspace/.version` back to `0.6.0`. Implementation:
   `tools/version.py.set_workspace_version(workspace, "0.6.0")`.

After revert, the framework files added by this release
(`tools/email/archive_hook.py`, the three new hook matchers in
`.claude/settings.json` + `.cursor/hooks.json`, the
`contracts/email-capture.md` section 8.1) remain on disk but are
inert until the IDE reloads them. To fully unwire capture-on-touch
post-revert, set
`workspace/_memory/config.yaml.preferences.privacy.archive_emails: false`
or remove the `PostToolUse` blocks from the two IDE config files.

The existing email archive at `workspace/_memory/email/` is not
affected by the revert.

## Notes

- The bridge accepts three response shapes: Gmail-API dict, MCP
  `CallToolResult` content array (`{"content": [{"type": "text",
  "text": "..."}]}`), and the gongrzhe MCP's text-only format. Schema
  drift in any of these is handled by silent fallback to no-op.
- The bridge writes a diagnostic log at `.tmp/email_archive_hook.log`
  on errors (gitignored). It never blocks the parent tool call.
- New config flag:
  `workspace/_memory/config.yaml.preferences.privacy.archive_emails`
  (default true). Set false to disable capture wholesale.
