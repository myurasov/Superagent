# Superagent daily-update skill

Use **today's date** in the user's local timezone (from `config.yaml.profile.timezone` or system).

Run the **MCP / CLI Preflight Protocol** (`contracts/mcp-preflight.md`) for every source whose row in `data-sources.yaml` has `capture_mode: scheduled` AND a schedule that includes `daily`.

## 1. Load configuration

Read `workspace/_memory/config.yaml`:

- `preferences.workspace_path` (default: `workspace`).
- `preferences.autonomous_enabled` / `daily_update_auto`.
- `preferences.surfacing.*` for surfacing windows.
- `profile.preferred_name` for greeting.

## 2. Load context

Read `workspace/_memory/context.yaml`:

- `last_check` (may be null on first run).
- `current_focus`, `pending_items`, `alerts` for continuity.
- `last_runs.daily_update` to detect "this is the first run today".

## 3. Run scheduled ingestors

For each row in `_memory/data-sources.yaml` with `enabled: true`, `capture_mode: scheduled`, and `schedule` matching today's cadence (per `contracts/ingestion.md` § 2.3):

1. Invoke `uv run python superagent/tools/ingest/<source>.py`.
2. Apply the per-source `max_items_per_run` cap.
3. Capture stdout summary; aggregate counts.

Print a one-line preamble at the top of the briefing:

> Ingested **N** new items from **K** sources (<source1>, <source2>, …) since your last check.

If an ingestor errored, surface a one-liner per failure under "Sources needing attention" at the bottom of the briefing — but never abort the briefing.

## 4. Today's calendar / appointments

Read `_memory/appointments.yaml`:

- Filter `status: scheduled` AND `start` is on today (local).
- Sort by `start` time.

For each event:
- Time, title, location, provider, for_member.
- Flag if `prep_done: false` and `prep_notes` non-empty.

If a calendar ingestor (gcal / icloud_calendar / outlook_calendar) is enabled but the event isn't in `appointments.yaml`, fall through to the live calendar source for the strictly-newer slice (`max(last_check, last_ingest) → end of today`). Capture-through per the ingestion contract.

Identify meetings that could benefit from prep. A meeting needs prep if any:
- External provider not in `contacts.yaml`.
- Sole-attendee meeting at an external location (likely a service provider visit).
- First-time meeting for a recurring series.
- Subject contains prep-heavy keywords ("annual physical", "tax prep", "annual review").

For each, emit a single line:

```
**Prep suggested:** {title} at {time} — {reason}. Run `appointments prep {id}` to assemble context.
```

## 5. Bills due today and this week

Read `_memory/bills.yaml`:

- **Overdue** — surface prominently.
- **Due today** — list (auto-pay items get one-line confirmation; manual items get an action prompt).
- **Due in 2–7 days** — short list.

For each: `{name} {amount} — {next_due} {auto-pay|manual}`. Format amounts with currency from `config.preferences.money.currency`.

If overdue count > 0, auto-create P0 tasks (one per overdue bill, marked `source: daily-update`). Update `todo.yaml` and the relevant `Domains/<domain>/status.md`.

## 6. Important dates (today + tomorrow + lookahead)

Read `_memory/important-dates.yaml`:

- **Today**: surface with greeting cues ("Mom turns 70 today — `important-dates draft-greeting <id>`?").
- **Tomorrow**: surface as a heads-up.
- **In `config.surfacing.important_dates_lookahead_days`**: list with date prefix.

Update each surfaced row's `surfaced_count`.

## 7. P0 / P1 tasks (open, due today, overdue)

Read `_memory/todo.yaml`:

- `priority in [P0, P1]` AND `status in [open, in_progress]` AND (`due_date <= today` OR `due_date is null`).
- Sort: overdue first (most-overdue first), then due-today, then no-date.

For each: `{id} {title} — {due_date or "no date"} ({related_domain})`.

If P0 count > 0, repeat the P0 list at the **top** of the briefing in a highlighted block (`## P0 — immediate attention`).

## 8. Health signals

Read `_memory/health-records.yaml`:

- **Refills due in next 7 days** (per-medication `next_refill_due`).
- **Active conditions** with `severity: severe` (always surface so they stay top of mind).
- **Symptoms** logged in last 24 hours (one line each).
- **Vital readings** outside normal range (e.g. BP > 140/90, glucose > 180) ingested in last 24 hours from any source.

If wearable ingestors are enabled, also surface:
- Last night's sleep score (whoop / oura / apple_health) — flag if < 60 / < 6 hours.
- Today's recovery / strain (whoop) — surface if "low recovery" today.
- Workout streak status if a fitness goal is active in `Domains/Hobbies/status.md`.

## 8b. Active project deadlines

Read `_memory/projects-index.yaml.projects[]` filtered by `status: active`:

- For each, compute `days_to_target = target_date - today`.
- Surface projects with `days_to_target <= config.preferences.projects.deadline_lookahead_days` (default 14) in a "Project deadlines" sub-block.
- For each, format: `{name} — {days_to_target} days, {open_task_count} open tasks ({path})`.
- For any active project with NO open tasks AND `days_to_target <= 7`, auto-create a P1 task: "Plan next steps for project <name>".
- Track project status flips since last daily-update (planning → active, active → paused, etc.) and surface as one-liners.

Also: surface any project marked `recurring: annual` whose target_date is in the next 30 days but is `status: planning` — likely needs to flip to `active`.

## 9. Maintenance windows (lookahead)

Read `_memory/assets-index.yaml`:

- For every asset with `maintenance` rows, compute `next_due` per row.
- Surface any with `next_due` in the next `config.surfacing.maintenance_lookahead_days`.

Read `_memory/documents-index.yaml`:

- Surface any document with `expires_at` in the next `config.surfacing.documents_lookahead_days`.

For each item, format: `{kind} for {asset_name or document_title} — due {next_due}`.

## 10. Inbox highlights (if email ingestor enabled)

If gmail / icloud_mail / outlook ingest ran in step 3:

- Read the latest interaction-log entries from the email ingestor (top 5–10 by Concierge-relevant heuristic: from a contact in `rolodex.md` of a domain with status: active, OR contains an action verb in the subject ("please review", "needs your", "RSVP"), OR is from a known important-domain sender).
- Format each: `{from} — {subject} ({domain hint})`.
- For each, suggest the obvious next action (`draft-email reply`, `add-appointment from-thread`, `add-bill from-thread`).

## 11. Alerts from other ingestors

Surface any single-row events captured by ingestors since last check that warrant attention:

- New large transaction (Plaid / Monarch ingestor flag).
- New auto-detected recurring charge (subscription audit candidate).
- Low balance on any account (`accounts-index.yaml.<acct>.low_balance_threshold` crossed).
- Vehicle mileage threshold crossed (Tesla ingestor → next-service threshold).
- Smart-home anomaly (Home Assistant ingestor → e.g. door open longer than usual).

Limit to 5 items max; the rest accumulate in next weekly-review.

## 12. Action signal nudges

Per `contracts/capture.md` § 7.2:

- Read `_memory/action-signals.yaml`.
- T = count of `target: tailor` AND `status: captured`.
- S = count of `target: superagent` AND `status: captured`.
- Include up to two one-liners under "Priority Items" (suppress if 0 or if the drain mechanism ran in last 24h):

  > **T unprocessed Supertailor signals** — run `supertailor-review` to digest.
  > **S pending Superagent workspace actions** — say "show pending workspace actions" to drain.

Output structure:

```

Ingested N new items from K sources since your last check.

## P0 — immediate attention

## Today

- HH:MM <event 2>
- Bills due: <list>
- Important dates: <today/tomorrow>

## This week
- Bills due in next 7 days: <list>
- Tasks due in next 3 days (P1): <list>
- Important dates in next <lookahead>: <list>
- Maintenance / documents in next <lookahead>: <list>

## Health signals
<refills, vital alerts, sleep / recovery, streak status>

## Inbox highlights (if applicable)
<top 5–10>

## Alerts
<step 11 items>

## Sources needing attention (if applicable)
<failed ingestors with one-line "what to fix" hints>

## Consider
<step 12 nudges>
```

Skip empty sections silently. Target ≤ 80 lines.

