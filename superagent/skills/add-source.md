# Superagent add-source skill

The Sources contract is in `contracts/sources.md`. This skill is the capture front-end. Remember: `Sources/` has a **user-defined layout**. The agent only owns `Sources/_cache/` and `Sources/README.md`. Everything else is the user's territory; this skill SUGGESTS paths but the user can always override.

## 0. Resolve the destination domain / project

The most common invocation is **"add this to <domain>"** or **"add X to <project>"** ÔÇö where `<domain>` / `<project>` is the user's natural-language reference to a row in `_memory/domains-index.yaml` / `_memory/projects-index.yaml`.

Resolution flow:

1. If the trigger phrase contains a domain name or slug, set `--to-domain <id>`.
2. If the trigger phrase contains a project name / slug, set `--to-project <id>`.
3. If both, set both ÔÇö the source will appear in both `sources.md` files.
4. If neither (e.g. "add this PDF to Sources" with no domain hint), infer from:
   - `category` once chosen (e.g. category `medical` Ôćĺ `--to-domain health`).
   - The active domain in `_memory/context.yaml.current_focus` if recent.
   - Otherwise ask the user.

This step ALWAYS resolves before file placement so the skill knows what to cross-link in step 3.

## 1. Branch on kind

Ask if not obvious from the trigger:

> "What kind of source? (1) document ÔÇö a local file I'll keep here; (2) reference ÔÇö a pointer to data that lives elsewhere (URL, MCP, CLI command, vault item, etc.)"

### Document mode

The user provides a path (or has just dropped a file in `Inbox/`). Steps:

1. Resolve the file. If in `Inbox/`, propose moving it.
2. **Category** (optional, free-text tag): ask the user, or infer from filename + content. Common values: `vehicles`, `taxes`, `medical`, `warranties`, `legal`, `insurance`, `property`, `education`, `pets`. The user can pick anything; categories are tags only ÔÇö they do not constrain the path.
3. **Title**: short, e.g. "Camry title 2018", "2024 federal return".
4. **Suggest a destination path** based on the category and any existing folder structure under `Sources/`. Examples:
   - `vehicles` Ôćĺ `Sources/vehicles/<vehicle-slug>/<filename>` (or just `Sources/vehicles/<filename>` if no `<vehicle-slug>`).
   - `taxes` Ôćĺ `Sources/taxes/<year>/<filename>`.
   - `medical` Ôćĺ `Sources/medical/<member-slug>/<filename>`.
   - `warranties` Ôćĺ `Sources/warranties/<asset-slug>/<filename>`.
   - `legal` Ôćĺ `Sources/legal/<filename>`.
   - `<custom>` Ôćĺ `Sources/<category>/<filename>`.
   Then **ask the user to confirm or override**: "I'll put this at `<suggested-path>` ÔÇö accept, or pick another path?"
5. **For member** (only if relevant): self / name / household.
6. **Cross-references** (optional):
   - `related_domain` (auto-suggest from category).
   - `related_project` (e.g. tax docs Ôćĺ `tax-<year>`).
   - `related_asset` (e.g. vehicle title Ôćĺ `<vehicle-slug>`).
7. **Sensitive?**: default false; flip true for medical, legal, accounts.
8. **Notes**: free text.

Move / copy the file to the chosen path. (Move from `Inbox/`; copy if from elsewhere.)

Optionally write a sibling `<filename>.ref.md` (Form A from ┬ž 15.3) carrying just the metadata:

```yaml
---
ref_version: 1
title: "<title>"
description: "<one-liner>"
kind: file
source: "<workspace-relative path to the document>"
ttl_minutes: 0          # local files are always fresh
sensitive: <bool>
related_domain: "<id>"
related_project: "<id>"
related_asset: "<id>"
added_by: "user"
added_at: <now>
tags: <list>
---

<notes>
```

(The sidecar `.ref.md` is OPTIONAL for documents ÔÇö the file IS the source. Skip if everything fits in `_memory/sources-index.yaml`.)

### Reference mode

The user wants to capture a pointer to external data (a portal that requires login, an MCP-fetchable item, a frequently-referenced URL).

1. **Title**: short (e.g. "Kaiser MyChart visit notes", "Fidelity 401k portal").
2. **Description**: one-line.
3. **Kind** (single-select):
   - `mcp` ÔÇö fetched via a configured MCP tool call.
   - `cli` ÔÇö fetched by running a shell command.
   - `url` ÔÇö fetched by HTTP GET (no auth).
   - `api` ÔÇö authenticated API call (token in vault).
   - `file` ÔÇö local file path (outside Sources/).
   - `vault` ÔÇö pulled from a password manager.
   - `manual` ÔÇö must be fetched by the user (e.g. portal with 2FA).
4. **Source identifier** (per the kind's format ÔÇö see `templates/sources/ref.md` for examples).
5. **TTL minutes**: default `1440` (24h); flip to `0` for "always fresh"; flip to `10080` (1 week) for slow-changing reference.
6. **Sensitive**: default false.
7. **Auth ref** (if `kind` requires auth): vault reference path.
8. **Cross-references**: same as document mode.
9. **Category** (free text, used as a tag only).

Suggest a destination path: `Sources/<category>/<short-slug>.ref.md` (or `Sources/<short-slug>.ref.md` if no category). **Ask the user to confirm or override.**

Write the canonical ref file to the chosen path using `superagent/templates/sources/ref.md` as template; substitute frontmatter values.

The body of the ref file is the user's notes. Prompt:

> "Anything else worth knowing about this reference? (Things that go in the body ÔÇö context, what to look for, how to interpret. The agent reads this BEFORE fetching, so a good note can avoid an unnecessary fetch.)"

The user is also free to author refs by hand in `.ref.txt` form (loose `Key: value` lines, or a bare URL on the first line, or a `1Password://...` URI). On first read the normalize-on-first-use prompt offers to canonicalize the file (see contracts/sources.md ┬ž 15.3 + `tools/sources_normalize.py`).

## 2. Place the file + refresh index

After placement (document or reference):

1. Run `python3 -m superagent.tools.sources_index refresh`. The walker picks up the new file and writes a row to `_memory/sources-index.yaml` with:
   - `id` derived from the workspace-relative path
   - `kind`, `title`, `path`, `category`
   - `present: true`, `read_count: 0`, `added: <now>`
   - any cross-references parsed from a sidecar / standalone ref's frontmatter
2. If you collected additional fields the agent should remember (notes, sensitive, related_*), update the row via `python3 -m superagent.tools.sources_index ... ` (or write the YAML in place ÔÇö refresh preserves them on subsequent runs).

The index is **derived from the filesystem**. The user can drop a file by hand from a shell at any time; the next read of `_memory/sources-index.yaml` triggers a refresh that picks it up. `add-source` is the convenience path; it does not own placement.

## 3. Cross-link to sources.md (the most important step)

**This is the contract that distinguishes `add-source` from a raw file copy.** Every source MUST be reachable from the human-readable per-domain / per-project catalogue:

### 3a. Per-domain `sources.md` write

If `related_domain` is set:

1. Open `workspace/Domains/<Domain>/sources.md`.
2. Resolve the right table:
   - `kind == document` Ôćĺ `## Documents`.
   - `kind == reference` Ôćĺ `## References (external pointers)`.
3. Append a new row using the row format defined in the template (`superagent/templates/domains/sources.md`):
   - For a document: `| <title> | <workspace-relative path> | <category> | <added-date> | <notes> |`.
   - For a reference: `| <title> | <ref path> | <kind> | <source identifier> | <notes> |`.
4. If the row already exists (same path), update notes only ÔÇö do not duplicate.
5. Update the `_Last updated:_` timestamp at top of the file.
6. Respect hand-edits: if the user has restructured the file (added custom sub-headings, regrouped rows, added a commentary paragraph), the skill performs a careful diff-and-merge, not a blind rewrite. Find the right table by header + columns, never by line number.

### 3b. Per-project `sources.md` write

If `related_project` is set: same routine as 3a, but against `workspace/Projects/<slug>/sources.md`. Plus the project may have its own `Projects/<slug>/Sources/` sub-tree ÔÇö if `--scope project` was passed, the actual file ALSO goes under `Projects/<slug>/Sources/<your-path>` (project-scoped) instead of the workspace-level `Sources/<your-path>`.

A source can have BOTH `related_domain` AND `related_project` set ÔÇö the row appears in both files.

### 3c. Asset cross-link

If `related_asset` is set: append the workspace-relative path to `_memory/assets-index.yaml.<asset>.receipts[]` (for receipts) or `.photos[]` (for photos that are vault-grade ÔÇö i.e. retained for insurance) or as a free-form `notes` reference (for other docs).

### 3d. Account cross-link

If `related_account` is set: append the path to `_memory/accounts-index.yaml.<account>.notes` as a single-line reference. (Per-account documents like statements typically belong to the Finances domain's `sources.md` AND get this account-level back-pointer.)

## 4. Confirm

```
Added <kind> "<title>" to <path>.
Cross-references: <list>.
Sensitive: <yes/no>.
Index id: <id>.
```

If kind is `reference`, also offer:

> "Want me to fetch this now to warm the cache? (yes / no ÔÇö defer to first use)"

On `yes`: invoke the matching ingestor / shell / HTTP fetch via `tools/sources_cache.py fetch <ref-path>`.

## 5. Sub-commands

- `add-source --replace <id> <new-file>` ÔÇö replace a source's content. The old
  file is renamed `<oldname>-superseded-<YYYY-MM-DD>.<ext>` next to it (history
  preserved); the new file takes the original name; the index row is updated
  via `sources_index refresh`.
- `add-source --untag <id> --from-domain <id>` ÔÇö remove the row from the
  domain's `sources.md`. The row is moved to `## Removed (history)` at the
  bottom of that file. The source itself stays in `Sources/`.
- `add-source --untag <id> --from-project <id>` ÔÇö same for projects.
- `add-source --move <id> --to-path <new-workspace-relative-path>` ÔÇö `mv` the
  file under `Sources/` and update the index. Use this when re-organizing
  the layout; the index id is regenerated from the new path.
