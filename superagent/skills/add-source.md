# Superagent add-source skill

The Sources contract is in `contracts/sources.md`. This skill is the capture front-end.

## 0. Resolve the destination domain / project

The most common invocation is **"add this to <domain>"** or **"add X to <project>"** — where `<domain>` / `<project>` is the user's natural-language reference to a row in `_memory/domains-index.yaml` / `_memory/projects-index.yaml`.

Resolution flow:

1. If the trigger phrase contains a domain name or slug, set `--to-domain <id>`.
2. If the trigger phrase contains a project name / slug, set `--to-project <id>`.
3. If both, set both — the source will appear in both `sources.md` files.
4. If neither (e.g. "add this PDF to Sources" with no domain hint), infer from:
   - `category` once chosen (e.g. category `medical` → `--to-domain health`).
   - The active domain in `_memory/context.yaml.current_focus` if recent.
   - Otherwise ask the user.

This step ALWAYS resolves before file placement so the skill knows what to cross-link in step 3.

## 1. Branch on kind

Ask if not obvious from the trigger:

> "What kind of source? (1) document — a local file I'll keep here; (2) reference — a pointer to data that lives elsewhere"

### Document mode

The user provides a path (or has just dropped a file in `Inbox/`). Steps:

1. Resolve the file. If in `Inbox/`, propose moving it.
2. **Category**: ask the user, or infer from filename + content:
   - `vehicles` (titles, registrations, insurance cards)
   - `taxes` (returns, W-2s, 1099s, receipts)
   - `medical` (records, lab results, vaccine cards)
   - `warranties` (appliance manuals, receipts)
   - `legal` (wills, trusts, POAs, advance directives)
   - `insurance` (policies, declarations pages)
   - `property` (deeds, mortgage docs, leases)
   - `education` (diplomas, certifications, school records)
   - `pets` (vet records, vaccinations)
   - `<custom>` (free text)
3. **Title**: short, e.g. "Camry title 2018", "2024 federal return".
4. **For member** (only if relevant): self / name / household.
5. **Cross-references** (optional):
   - `related_domain` (auto-suggest from category).
   - `related_project` (e.g. tax docs → `tax-<year>`).
   - `related_asset` (e.g. vehicle title → `<vehicle-slug>`).
6. **Sensitive?**: default false; flip true for medical, legal, accounts.
7. **Notes**: free text.
8. Auto-derive **id** = `doc-<category>-<short-slug>` (e.g. `doc-vehicles-camry-title-2018`).

Move / copy the file to `workspace/Sources/documents/<category>/<filename>`. (Move from `Inbox/`; copy if from elsewhere.)

Optionally write a sibling `.ref.md` carrying just the metadata:

```yaml
ref_version: 1
title: "<title>"
description: "<one-liner>"
kind: file
source: "Sources/documents/<category>/<filename>"
ttl_minutes: 0          # local files are always fresh
sensitive: <bool>
related_domain: "<id>"
related_project: "<id>"
related_asset: "<id>"
added_by: "user"
added_at: <now>
category: "<category>"
tags: <list>
```

(The .ref.md is OPTIONAL for documents — the file IS the source. Skip if metadata is in `_memory/sources-index.yaml`.)

### Reference mode

The user wants to capture a pointer to external data (a portal that requires login, an MCP-fetchable item, a frequently-referenced URL).

1. **Title**: short (e.g. "Kaiser MyChart visit notes", "Fidelity 401k portal").
2. **Description**: one-line.
3. **Kind** (single-select):
   - `mcp` — fetched via a configured MCP tool call.
   - `cli` — fetched by running a shell command.
   - `url` — fetched by HTTP GET (no auth).
   - `api` — authenticated API call (token in vault).
   - `file` — local file path (outside Sources/).
   - `vault` — pulled from a password manager.
   - `manual` — must be fetched by the user (e.g. portal with 2FA).
4. **Source identifier** (per the kind's format — see `templates/sources/ref.md` for examples).
5. **TTL minutes**: default `1440` (24h); flip to `0` for "always fresh"; flip to `10080` (1 week) for slow-changing reference.
6. **Sensitive**: default false.
7. **Auth ref** (if `kind` requires auth): vault reference path.
8. **Cross-references**: same as document mode.
9. **Category** (free text or one of the known categories).

Auto-derive **id** = `ref-<category>-<short-slug>`.

Write to `workspace/Sources/references/<category>/<short-slug>.ref.md` using `superagent/templates/sources/ref.md` as template; substitute frontmatter values.

The body of the `.ref.md` is the user's notes. Prompt:

> "Anything else worth knowing about this reference? (Things that go in the body — context, what to look for, how to interpret. The agent reads this BEFORE fetching, so a good note can avoid an unnecessary fetch.)"

## 2. Update sources-index

Append a row to `_memory/sources-index.yaml.sources[]`:

```yaml
- id: "<id>"
  kind: "document" | "reference"
  title: "<title>"
  path: "<workspace-relative>"
  category: "<category>"
  related_domain: "<id or null>"
  related_project: "<id or null>"
  related_asset: "<id or null>"
  sensitive: <bool>
  added: <now>
  last_accessed: null
  read_count: 0
  tags: <list>
  notes: "<notes>"
```

## 3. Cross-link to sources.md (the most important step)

**This is the contract that distinguishes `add-source` from a raw file copy.** Every source MUST be reachable from the human-readable per-domain / per-project catalogue:

### 3a. Per-domain `sources.md` write

If `related_domain` is set:

1. Open `workspace/Domains/<Domain>/sources.md`.
2. Resolve the right table:
   - `kind == document` → `## Documents`.
   - `kind == reference` → `## References (external pointers)`.
3. Append a new row using the row format defined in the template (`superagent/templates/domains/sources.md`):
   - For a document: `| <title> | <workspace-relative path> | <category> | <added-date> | <notes> |`.
   - For a reference: `| <title> | <ref path> | <kind> | <source identifier> | <notes> |`.
4. If the row already exists (same path), update notes only — do not duplicate.
5. Update the `_Last updated:_` timestamp at top of the file.
6. Respect hand-edits: if the user has restructured the file (added custom sub-headings, regrouped rows, added a commentary paragraph), the skill performs a careful diff-and-merge, not a blind rewrite. Find the right table by header + columns, never by line number.

### 3b. Per-project `sources.md` write

If `related_project` is set: same routine as 3a, but against `workspace/Projects/<slug>/sources.md`. Plus the project may have its own `Projects/<slug>/Sources/` sub-tree — if `--scope project` was passed, the actual file ALSO goes under `Projects/<slug>/Sources/documents/<category>/<...>` (project-scoped) instead of the workspace-level `Sources/documents/<category>/`.

A source can have BOTH `related_domain` AND `related_project` set — the row appears in both files.

### 3c. Asset cross-link

If `related_asset` is set: append the workspace-relative path to `_memory/assets-index.yaml.<asset>.receipts[]` (for receipts) or `.photos[]` (for photos that are vault-grade — i.e. retained for insurance) or as a free-form `notes` reference (for other docs).

### 3d. Account cross-link

If `related_account` is set: append the path to `_memory/accounts-index.yaml.<account>.notes` as a single-line reference. (Per-account documents like statements typically belong to the Finance domain's `sources.md` AND get this account-level back-pointer.)

If kind is `reference`, also offer:

On `yes`: invoke the matching ingestor / shell / HTTP fetch via `tools/sources_cache.py fetch <id>`.

## 5. Immutability reminder

End with:

```
Note: Sources/<kind>/ is immutable. Skills won't delete or overwrite this.
To replace it, run: add-source --replace <id>
To remove it: do it manually with `rm` from a shell.
   (The pointer in <Domain>/sources.md will be moved to '## Removed (history)'
    on next sync, but the file itself is yours to manage.)
```

## 5b. Sub-commands

- `add-source --replace <id> <new-file>` — replace a source's content. The old
  file is renamed `<oldname>-superseded-<YYYY-MM-DD>.<ext>` (keep history),
  the new file takes the original name, and the `sources.md` row is updated.
- `add-source --untag <id> --from-domain <id>` — remove the row from the
  domain's `sources.md`. The row is moved to `## Removed (history)` at the
  bottom of that file. The source itself stays in `Sources/`.
- `add-source --untag <id> --from-project <id>` — same for projects.
- `add-source --move <id> --to-domain <new-id>` — change the cross-link.
