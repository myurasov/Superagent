# Superagent migrate skill

The **sole** entry point for applying or reverting framework version
migrations on a workspace. Implements `contracts/versioning.md` Â§ 4.

User priorities, in order:

1. **Keep the workspace working.** A failed migration must leave the
   workspace usable â€” at the highest successfully-migrated version,
   never half-applied.
2. **Be reversible.** Every migration in this directory must be
   revertible by default; refuse to migrate further if a step is marked
   `revertible: false` and the user has not explicitly accepted the
   trade-off.
3. **Never surprise.** Always show the chain and ask before applying.
   Dry-run is always available.

## 1. Resolve mode

The skill has three modes, picked by the trigger phrase:

- **forward (default)** â€” `migrate`, `upgrade`, "version mismatch detected"
- **dry-run** â€” user passed `--dry-run` or chose `dry-run` in the
  confirmation prompt
- **revert** â€” user said `revert`, `downgrade`, or passed
  `revert [--to <version>]`

For `forward` and `dry-run`, continue to step 2. For `revert`, jump to step 9.

## 2. Read state

Run `uv run python -m superagent.tools.version check`. Capture:

- the framework's current version (`current_version()`)
- the workspace's current version (`workspace_version()`, defaults
  `0.1.0` if `.version` is missing)
- the exit code (0 = match, 1 = migration needed, 2 = downgrade)

If exit code is **0**: report `Already on <version> â€” nothing to do.`
and exit. (Skill done; do not log.)

If exit code is **2**: jump to step 8 (downgrade scenario).

If exit code is **1**: continue.

## 3. Compute chain + classify

Run `uv run python -m superagent.tools.version chain` to print the chain.
Parse the result.

If the message says "PATCH-only", continue to step 4 (silent bump).
Otherwise (one or more MINOR/MAJOR steps), continue to step 5.

## 4. PATCH-only auto-bump

Bump silently per `contracts/versioning.md` Â§ 4 step 8:

1. `uv run python -m superagent.tools.version set <current_version>`
2. Append a row to `workspace/_memory/interaction-log.yaml`:
   ```yaml

     ts: <iso datetime>
     skill: superagent-migrate
     action: patch_auto_bump
     from_version: <ws_version>
     to_version: <current_version>
     migrations_applied: 0
     notes: "PATCH-only delta; .version advanced silently per contracts/versioning.md Â§ 4 step 8"
   ```
3. Report: `Advanced workspace .version <ws> -> <cur> (PATCH-only).`
   Exit.

## 5. Show plan + confirm

Print to the user:

```
Migration plan: <ws> -> <cur>
N steps, estimated total <sum of estimated_duration> minutes.

  1. <from> -> <to>: <title>          [breaking|revertible:false]
  2. <from> -> <to>: <title>
  ...

Each step writes to:
  - workspace/.version
  - <touches list, deduped>

Each migration's instructions live at:
  superagent/migrations/<to>.md
```

Then `AskQuestion` (single question, three options):

- `proceed` â€” apply the chain step by step
- `dry-run` â€” print each step's `## Migrate` body without applying
- `cancel` â€” exit without changes

If the chain contains any step with `revertible: false`, **add a
fourth option** `proceed (accept non-revertible step)` and require the
user to pick that one explicitly to continue. The other `proceed`
option becomes disabled â€” refuse to advance past a non-revertible step
without the explicit acceptance.

## 6. Apply chain (forward)

For each migration in the chain, in order:

### 6.1 Read the migration file

`Read superagent/migrations/<to_version>.md` in full. Extract:
- `## Summary`
- `## Pre-flight checks` (the bullet list)
- `## Migrate` (numbered steps)
- `## Validate` (bullet list)
- `## Revert` (numbered steps; needed if a later step in this run fails)

### 6.2 Run pre-flight checks

For each `- [ ]` bullet: convert to a verifiable test (file exists, YAML
parses, key present, etc.) and run it. On any failure:

- Print: `Pre-flight failed for <to_version>: <which check>.`
- Print: `Workspace remains at <last_successful_version>.`
- Print the revert command if anything was applied earlier in the run.
- Log the failure to `interaction-log.yaml` with `action: migrate_aborted`.
- **Exit immediately**. Do not continue to the next migration.

### 6.3 Execute the Migrate steps

For each numbered step in `## Migrate`:

- If the step says "run `uv run python superagent/migrations/<v>/migrate.py`",
  invoke that script. Pass `--workspace <path>` and `--dry-run` only when
  in dry-run mode.
- If the step is a file-system or YAML edit, perform it directly using
  the appropriate tool (do NOT shell out to `sed`/`awk` per `AGENTS.md`).
- If the step references a helper from `helper_scripts.migrate` in
  the frontmatter, prefer that script over manual interpretation when
  both are present â€” the script is authoritative.

In **dry-run** mode, print what each step *would* do without performing
it.

### 6.4 Validate

For each `- [ ]` bullet in `## Validate`: run the verification. On any
failure:

- Print: `Validation failed for <to_version>: <which check>.`
- Print: `Workspace partially migrated. Run \`migrate revert\` to roll
  back this step.` (Do NOT auto-revert â€” leave the user in control.)
- Log to `interaction-log.yaml` with `action: migrate_validation_failed`.
- **Exit**.

### 6.5 Update `.version` + log

On success of a single step:

1. `uv run python -m superagent.tools.version set <to_version>`
2. Append to `workspace/_memory/interaction-log.yaml`:
   ```yaml
   - id: <new id>
     ts: <iso datetime>
     skill: superagent-migrate
     action: migrate_step_applied
     from_version: <step.from_version>
     to_version: <step.to_version>
     migration_file: migrations/<to_version>.md
     breaking: <step.breaking>
     revertible: <step.revertible>
     duration_seconds: <measured>
     notes: ""
   ```

Move on to the next migration in the chain.

## 7. Final summary

After the chain completes (all steps successful):

```
Migrated workspace <ws> -> <cur> in <N> step(s), <total duration>.

If anything looks off, run:
  migrate revert --to <ws>
to roll back to the prior state.
```

Append a final summary row to `interaction-log.yaml` with
`action: migrate_chain_completed`, `migrations_applied: <N>`,
`from_version: <ws>`, `to_version: <cur>`.

Skill done.

## 8. Downgrade scenario

(Reached when `tools/version.py check` returns exit code 2.)

The workspace's `.version` is HIGHER than the framework's. Pull the
matching framework, or revert the workspace.

`AskQuestion` (single question, three options):

- `upgrade-framework` â€” recommended; print the git command to fetch the
  matching framework version (the user runs it; the skill does not pull
  for them per `AGENTS.md` Â§ "Privacy and data location").
- `revert-workspace` â€” runs the revert chain down to the framework's
  version. Requires every step to be `revertible: true`; refuses
  otherwise with a clear list of the offending steps.
- `cancel` â€” leave both alone.

On `upgrade-framework`: print
```
Run: git fetch && git checkout v<workspace_version>
```
and exit.

On `revert-workspace`: jump to step 9 with `--to <framework_version>`.

On `cancel`: exit with a one-line warning that subsequent skills may
misbehave because the workspace is on a newer version than the
framework knows about.

## 9. Revert mode

Triggered by `revert`, `downgrade`, or routed-from-step-8.

### 9.1 Resolve target

If user passed `--to <version>`: target is that version.
Otherwise: target is the immediately-previous migration's
`from_version` (peel off the most recent migration only).

### 9.2 Compute revert chain

Reverse-walk the manifest from the workspace's current version down to
the target. If ANY step in this range has `revertible: false`:

- List the offending steps and the data they irreversibly modified.
- Print the recommended manual restore (e.g. "restore from your
  most recent backup of `workspace/`").
- Refuse to proceed. Exit.

### 9.3 Show plan + confirm

Same shape as step 5 but in reverse order:

```
Revert plan: <ws> -> <target>
N steps, estimated total <sum> minutes.

  1. <to> -> <from>: <title>
  ...

Each step's revert instructions live at:
  superagent/migrations/<to>.md (## Revert section)
```

`AskQuestion`: `proceed` / `dry-run` / `cancel`.

### 9.4 Apply revert chain

For each migration in REVERSE order:

1. Read `superagent/migrations/<to_version>.md`.
2. Execute the `## Revert` numbered steps (or invoke
   `helper_scripts.revert` if defined in frontmatter).
3. After the revert completes successfully, set `.version` to the
   migration's `from_version` (the version we just reverted *to*).
4. Append `action: migrate_step_reverted` row to `interaction-log.yaml`
   with the from/to versions inverted vs forward migration.

On any revert failure, halt the chain. Surface the exact state of
`workspace/.version` and the list of remaining-to-revert steps.

### 9.5 Final revert summary

```
Reverted workspace <starting_ws> -> <ending_target> in <N> step(s).
```

Append `action: revert_chain_completed` row.

Skill done.
