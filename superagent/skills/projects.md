# Superagent projects skill

The full Projects contract is in `procedures.md` § 16.

## 1. Branch on intent

### List (default — `projects`)

1. Read `_memory/projects-index.yaml.projects[]`.
2. Filter by status (default: not in `[completed, cancelled, archived]`).
3. Group by status (`active` first, then `planning`, then `paused`).
4. Within each status, sort by `target_date` ascending (null last).
5. For each, render one line:

   ```
   <slug>  [<status>]  <name>
       Goal: <goal>
       Target: <target_date> (<N> days)  Domains: <related_domains>
       Open tasks: <count from todo.yaml>
   ```

6. End with a one-line: `Total active: <N>; planning: <K>; paused: <M>; recently completed (90d): <P>.`

Filters:
- `--status <status>` to filter.
- `--domain <slug>` to show only projects linked to a domain.
- `--all` to include completed / archived.

### Show (`projects show <slug>` or "show me the X project")

1. Read the row from `_memory/projects-index.yaml`.
2. Read `Projects/<slug>/info.md`, `status.md`, `history.md` (last 5 H4 entries), `rolodex.md` (just count of contacts).
3. Render a structured brief:

   ```
   # Project: <name>  [<status>]

   Goal: <goal>
   Target: <target_date> (N days remaining)
   Started: <start_date>
   Related domains: <list>
   Recurring: <recurring>
   Stakeholders: <count>
   Budget: <planned/spent/variance> (if any)

   ## Recent history
   <last 5 H4 entries summarized>

   ## Status
   <RAG line>
   <Recent Progress bullets>
   <Active Blockers bullets>
   <Next Steps bullets>

   ## Burn-down (if applicable)
   <checked>/<total> success criteria — <days> days to target

   ## Open tasks
   <table from todo.yaml filtered by related_project>
   ```

### Complete (`projects complete <slug>` or "I just finished X")

1. Resolve project by slug or fuzzy name match.
2. Verify success criteria — list each, ask if all met. If some are not met:

   > "Success criteria not met: <list>. Complete anyway, or update them first?"

3. On confirm:
   - `status: completed`, `completed_date: now`.
   - Append H4 to `history.md`: `#### <YYYY-MM-DD> — Project completed`.
   - For each `related_domain`, append H4 to `Domains/<Domain>/history.md`: `#### <YYYY-MM-DD> — Project completed: <name>`.
   - Mark any open tasks for this project as `cancelled` with note "project completed", OR ask user to mark each `done`.

4. **If `recurring` ≠ `none`**, auto-create the next-cycle project (per `procedures.md` § 16.7):
   - Generate next slug (e.g. `tax-2025` → `tax-2026`).
   - Inherit `name` (year-substituted), `scope`, `success_criteria`, `deliverables`, `stakeholders`, `related_domains`, `recurring`.
   - Compute next `target_date` (annual = +1y; quarterly = +3mo; etc.).
   - `status: planning`; `start_date` = next cycle's natural start.
   - Add `## Prior cycles` link to the just-completed instance's path.
   - Confirm with user before writing.

5. Surface a "lessons learned" prompt:

   > "What went well? What would you do differently next time?"

   Capture as personal-signal AND as an H4 entry in the just-completed `history.md`.

### Pause (`projects pause <slug>`)

1. `status: paused`. Append H4 to `history.md` with reason.
2. Suppressed from active surfacing in cadences.

### Resume (`projects resume <slug>`)

1. `status: active`. Append H4 to `history.md`.
2. Re-rendered in next cadence.

### Cancel (`projects cancel <slug>`)

1. `status: cancelled`. Prompt for reason; capture in `history.md`.
2. Open tasks for this project: ask whether to cancel, reassign, or leave unscoped.

### Archive (`projects archive <slug>`)

(Usually called by `doctor` after `archive_after_days`; can be called directly.)

1. Verify `status` in `[completed, cancelled]`. If `active`, refuse.
2. Move `Projects/<slug>/` → `Archive/<YYYY-MM>/Projects/<slug>/`.
3. Update `_memory/projects-index.yaml`:
   - Move row from `projects[]` to `archived[]`.
   - Update `path` to the new location.

### Burn-down (`projects burndown <slug>`)

Render the per-project burn-down chart (text-based):

```
Project: <name>  Target: <date> (<N> days)

Success criteria:
  [x] <criterion 1>          (completed <date>)
  [x] <criterion 2>          (completed <date>)
  [ ] <criterion 3>
  [ ] <criterion 4>
  [ ] <criterion 5>

Progress: ███████░░░░░░░░ 2/5 (40%)
Days elapsed: <N> / <total>
On pace: <yes / behind / ahead>
```

## 2. Sync `status.md`

After any state change, regenerate `Projects/<slug>/status.md`:

- `### Current Status (RAG)`: derived from days-to-target vs success-criteria progress (`Green` if on pace, `Amber` if slipping, `Red` if late or blocked).
- `### Recent Progress`: last 5 entries from `history.md` summarized.
- `### Active Blockers`: pulled from `history.md` H4 entries tagged "blocker" / "blocked".
- `### Next Steps`: top 3 P1/P2 open tasks from `todo.yaml` for this project.
- `### Burn-down`: per the criteria above.
- `## Open` and `## Done`: regenerated from `todo.yaml` via `tools/render_status.py --scope project:<slug>`.

## 3. Cross-link domains

When a project's `status` flips, also append a one-bullet update to each `related_domain`'s `Domains/<Domain>/status.md` § Next Steps and an H4 entry to its `history.md`.
