# Superagent todo skill

Follow these steps in order; skip steps that don't apply.

## 1. Load configuration

Read `workspace/_memory/config.yaml` to resolve the workspace root.

## 2. Load the task file

Read `workspace/_memory/todo.yaml`. If missing, initialize from `superagent/templates/memory/todo.yaml` (preserve `schema_version`), then continue.

## 3. Branch on user intent

### Add a task

When the user asks to add a task:

1. Collect or infer:
   - **title** (mandatory; ≤ 80 chars).
   - **description** (optional; details, links, acceptance criteria).
   - **priority** (P0 / P1 / P2 / P3). Default P2. Use heuristics:
     - "today" / "urgent" / "now" / "ASAP" → P0.
     - "this week" / "by Friday" / specific date in next 3 days → P1.
     - "this month" / no date / generic action → P2.
     - "someday" / "maybe" / "future" → P3.
   - **due_date** (ISO 8601 date or datetime; null if not scheduled).
   - **related_domain** (slug from `domains-index.yaml`; infer from task content if obvious).
   - **related_asset / related_account / related_appointment / related_bill** (optional cross-references).
   - **tags** (list of short labels).
   - **source** = `"user"` for user-added, otherwise the skill / ingestor that created it.
2. Generate **id** = `task-<YYYYMMDD>-<NNN>` where NNN is the next sequential 3-digit number for today.
3. Set **status** = `open`, **created** = now (ISO 8601 with TZ).
4. Append the new record.

### List tasks

When the user asks to list / show tasks:

1. Filter to relevant statuses (default: `open` + `in_progress`; include `done` only on request).
2. **Group** by **priority** (P0 first).
3. Within each group, sort by **due_date** (soonest first; null last).
4. **Overdue**: compute today in the user's timezone. Tasks with `due_date < today` are overdue — list them first within their priority group with an `OVERDUE` prefix.
5. If any P0 tasks exist, surface them prominently at the top.
6. Filter further per user request (`--domain Health`, `--project tax-2026`, `--asset blue-camry`, `--tag taxes`).

### Complete a task

When the user asks to mark complete / close a task:

1. Resolve by **id** or unambiguous title match. Confirm if multiple match.
2. Set `status: done`, `completed_date: now`.

### Update a task

When the user asks to change a task:

1. Resolve by id or unambiguous match.
2. Modify only the **specified fields**. Do not clear unspecified fields unless the user asked.

### No specific request

When invoked without a clear add/list/complete/update action:

1. Compute:
   - **X** = open tasks (count).
   - **Y** = open tasks overdue.
   - **Z** = tasks completed this week.
2. Print this summary and ask what they want to do next.

## 4. Persist

1. Write changes back to `workspace/_memory/todo.yaml`. Preserve formatting, comments, and schema_version.

## 5. Sync `status.md` / `todo.md` files

After any add / complete / update operation, update the relevant scoped views per the sync contract (`contracts/task-management.md` § 5.1):

1. For each task touched, resolve `related_project` then `related_domain`:
   - `related_project` set → rewrite `workspace/Projects/<slug>/status.md` (priority/due-date sort; per-project burn-down updated).
   - `related_domain` set (and no project) → rewrite `workspace/Domains/<Domain>/status.md`.
   - Neither set → update `workspace/todo.md` (the workspace-level cross-cutting view).
   - When BOTH set: render in the project's status.md (project takes precedence as the "doing" surface), AND surface a one-bullet summary in the related domain's `status.md` § Next Steps.
2. Format as markdown tables, grouped by priority, with a separate `Done` block at the bottom (per `templates/todo.md`):

   ```
   ## P0 — Today / Urgent

   | ID | Task | Due | Domain |
   |----|------|-----|--------|
   | task-NNN | Title text | Mon DD | <domain-id> |
   ```

   - Within each priority block, sort by due date (soonest first; no-date rows last).
   - Use `—` for missing due dates / domains.
   - Omit empty priority blocks.
3. Update the `_Last updated:_` timestamp at the top of each modified file.
4. If a `status.md` doesn't exist for a scope, create from `superagent/templates/domains/status.md`.

## 6. P0 highlighting

If any P0 tasks exist and are still open after the operation:

1. End the response with a highly visible block:

   ```
   ## P0 — immediate attention
   - **task-NNN** Title — due Mon DD
   ```

2. This applies even when the user only asked for a summary or a partial update.

```yaml
  related_domain: <if applicable>
  action_items: []
```
