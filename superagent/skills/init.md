# Superagent initialization skill

Follow these steps in order. Use the repo root as the working directory unless `workspace/_memory/config.yaml` already defines a different `preferences.workspace_path`.

## 1. Welcome message

Open with a brief 4-line welcome:

- **Superagent** is a personal-life AI assistant — it shadows you across bills, health, home, family, and the rest of your life so the admin overhead stops crowding out the work that matters.
- This setup takes about **5 minutes** for the quick-start path: a 3-question questionnaire, scaffold a folder structure, and try a couple of capture commands. No data sources required.
- Optional **heavy-import path** (after the quick-start): probe for MCPs and CLI tools that are already set up on your machine, and enable the ones you want. You can also run `ingest --setup` later.
- Everything stays **local** in `workspace/` (gitignored). No telemetry, no cloud sync, no remote write.

Then immediately proceed to step 2.

## 2. Quick-start questionnaire (mandatory)

Use the `AskQuestion` tool. **One batch**, three questions, takes a minute.

**Question 1 — "What should I call you?"** (text-input style — present a couple of common options and a "type a name" escape):

Options:
- "Use my Git name"
- "Use my macOS account name"
- "Let me type it"

(If the user picks "Let me type it", prompt for the value in the next message.)

**Question 2 — "Who is in your household?"** (allow_multiple: true)

Options:
- Just me
- Spouse / partner
- Kids
- Parents (live with you)
- Roommates
- Dog(s)
- Cat(s)
- Other pet(s)
- Other (I'll describe)

**Question 3 — "What's the most painful personal-admin friction RIGHT NOW?"** (allow_multiple: true)

Options:
- Bills — I miss due dates, forget recurring charges, or fight late fees
- Subscriptions — I have no idea what I'm paying for
- Health — I can never remember when I last saw the dentist / had labs / refilled X
- Important dates — birthdays, anniversaries, recurring deadlines slip
- Appointments — booking, prep, follow-up takes too much energy
- Documents — I can never find the passport / title / insurance card when I need it
- Home maintenance — filters, HVAC, gutters, anything seasonal falls through
- Vehicles — service intervals, registration, insurance renewals slip
- Pets — vet schedule, meds, food rotation
- Family logistics — kids' schedules, school stuff, parent care
- Email triage — too much, hard to prioritize
- Finances — I don't know where my money goes
- Hobbies / fitness — I lose track of progress
- Just feeling overwhelmed — I want one place that knows everything

Capture answers — they drive (a) seeded `_memory/insights.yaml` rows, (b) which capture skill the walkthrough demos in step 7, (c) which optional data sources to recommend with priority in step 5.

**Question 4 — "How far back should spreadsheet exports reach?"** (single-select)

Per the workbook subsystem (see `skills/workbooks.md`), Superagent renders
human-friendly `.xlsx` projections of your structured data — Bills,
Transactions, Vet Visits, Service History, etc. Time-series sheets respect
this window; reference sheets (Accounts, Properties, Certifications) ignore
it.

Options:
- 5 years
- **10 years** (default — covers most personal-tax + net-worth cases)
- 20 years
- All available history (no truncation)

Captured into `_memory/config.yaml.preferences.workbooks.history_window_years`.

## 3. Scaffold the workspace

Run `uv run python superagent/tools/workspace_init.py`. The script:

1. Creates `workspace/` if missing.
2. Creates `workspace/_memory/` and copies every YAML template from `superagent/templates/memory/` into it (only if the destination file doesn't exist; never overwrite user data). The shipped `domains-index.yaml` template **registers** the 13 default domains so capture skills can route to them.
3. Creates `workspace/Domains/` (the directory only — per-domain folders are LAZY per `contracts/domains-and-assets.md` § 6.4a; they materialize the first time data lands for a given domain). Drops the explanatory `Domains/README.md` template explaining the convention.
4. Creates `workspace/Projects/` (empty; populated as the user adds projects via `add-project`).
5. Creates `workspace/Sources/` with `documents/`, `references/`, `_cache/` subdirectories (per the immutability + caching contract in `contracts/sources.md`).
6. Creates `workspace/Inbox/`, `workspace/Outbox/`, `workspace/Archive/`, and drops the explanatory README from `superagent/templates/folder-readmes/` into each.
7. Creates `workspace/_custom/` with empty `rules/`, `skills/`, `agents/`, `templates/`, `tools/` subdirectories (per the custom-overlay contract).
8. Creates the workspace-level `workspace/todo.md` from `superagent/templates/todo.md` (with no rows yet).
9. Records workspace-init timestamp in `_memory/interaction-log.yaml`.

Read back the script's stdout to the user as confirmation; surface any errors.

## 4. Populate `config.yaml` from the questionnaire

Open `workspace/_memory/config.yaml` and fill in:

- `last_updated` → now (ISO 8601 with TZ).
- `profile.preferred_name` from Q1 (with fallback to git or macOS user name if "Use my X name" was picked).
- `profile.timezone` from system (`date +%Z` or equivalent).
- `profile.household_composition` from Q2 (concatenated short text, e.g. "spouse + 2 kids + 1 dog").
- `preferences.pain_points` from Q3 (the option ids).
- `preferences.workbooks.history_window_years` from Q4 (5 / 10 / 20 / null for "all").

If Q3 included specific pain points, also seed `_memory/insights.yaml` with one row per pain point mapping it to the recommended skill (see § 8 mapping reference).

## 5. Probe for available data sources (optional, opt-in)

Ask the user (single question):

> "Want me to look for data sources I could ingest from? I'll check what's installed on this machine and ask before enabling each one. (No data is read in this step — only presence checks.) **yes / no / later**"

If the user picks **no** or **later**, skip to step 6.

If the user picks **yes**:

1. Read `workspace/_memory/data-sources.yaml` (just-created with all sources `enabled: false`).
2. For each source in the catalogue (see `docs/data-sources.md`), run a presence probe — the lightweight check documented for that source (see also `tools/ingest/<source>.py` `probe()` function). Examples:
   - `gmail`: check `~/.config/gmail-mcp/` for OAuth tokens OR check whether a `google_workspace_mcp` server is registered in any of the running IDEs' MCP configs.
   - `icloud_calendar`: check macOS / iCloud login state via `defaults read MobileMeAccounts` (no actual data read).
   - `apple_health`: look for an existing `~/.healthsync/healthsync.db` OR check if there's an `Apple Health` MCP server registered.
   - `apple_reminders`: `which rem` AND attempt `rem list --json | head -1` (fails silently if no Reminders permission).
   - `plaid`: check for `~/.config/plaid-cli/config.json`.
   - `monarch`: check for the `monarch-cli` binary on `$PATH`.
   - `strava`, `whoop`, `garmin`, `oura`: check for any registered MCP server matching the source name OR vendor-specific config files in `~/.config/`.
   - `home_assistant`: check for a Home Assistant MCP server registration OR `HOMEASSISTANT_URL` env var.
   - `tesla`: check for the Tesla MCP server.
   - `obsidian`: check for `~/Library/Application Support/obsidian/` (macOS) OR for an Obsidian MCP server registration. If found, ask the user for the vault path.
   - `notion`: check for `NOTION_TOKEN` env var or a Notion MCP server registration.
   - `imessage`: check `~/Library/Messages/chat.db` exists AND that we have permission (a small read attempt).
   - `photos`: `which exiftool`.
   - `gmaps_timeline`: ask the user — there's no automatic detection (it's a Google Takeout export the user has to download).
3. Build a presence summary table. Mark each source as `available` / `not detected` / `needs setup`.
4. **Prioritize** sources by:
   - The user's pain points from Q3 (e.g. "Subscriptions" → recommend Plaid / Monarch / Gmail; "Important dates" → recommend Gmail / Calendar; "Health" → recommend Apple Health / wearables).
   - Likely value-per-minute-of-setup (start with email + calendar, which give a huge lift for ~5 minutes of OAuth).
5. **For each available source**, ask the user:

   > "I found **<source>** available. Enabling it will let Superagent pull <one-line summary of what gets ingested>. First-run will fetch the **last <recency_window_days> days** with a max of **<max_items_per_run> items**. **Enable now / enable later / skip**?"

   On "enable now": flip `data-sources.yaml.<source>.enabled = true`, set `capture_mode = scheduled`, and run a first ingest (see step 5b).

6. **For each not-detected source the user mentioned interest in** (Q3 mentioned "subscriptions" but no finance source detected), end with:

   > "**<source>** isn't installed yet. Setup takes ~<N> minutes — see `superagent/docs/data-sources.md#<source>` for the install steps. I can pick this up next time you run `ingest --setup`."

### 5b. First-run ingestion (per source the user enabled)

For each source enabled in step 5:

1. Invoke `uv run python superagent/tools/ingest/<source>.py --first-run` with the conservative `recency_window_days` from `data-sources.yaml`.
2. Surface live progress (every 10 items or 5 seconds, whichever first): "<source>: 47 items pulled, 41 inserted, 6 deduped …".
3. On completion, print a one-line summary: "<source>: <N> items ingested, <K> domains touched, took <T>s."
4. On failure, log the error and keep going — no source's failure aborts the rest.

The full first-run might take a couple of minutes total if many sources were enabled. Print a running tally.

## 6. Update `model-context.yaml`

Seed initial entries:

- `communication.style`: best guess from how the user has been chatting in the init session (e.g. "terse" if they replied "yes" to the questionnaire questions; "thorough" if they typed paragraphs). It's fine to leave blank if unsure.
- `household_context.pets`, `household_context.kids`, `household_context.partner`: from Q2 (note: just placeholders; the user can fill names later via `add-contact`).
- `working_patterns.schedule`: leave blank (the daily-update skill learns this over time).
- `last_updated` → now.

Append a one-line session summary to `sessions[]`: `{ date: <today>, summary: "Initial setup. <pain points>. <enabled sources>." }`.
